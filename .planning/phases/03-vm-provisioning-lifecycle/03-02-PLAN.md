Create provisioning service:

1. **src/lib/provisioning/processor.ts** — processProvisioningJobs():
   - Find ProvisioningJob with status 'pending', limit 5
   - For each: call processOneJob(job)
   - Check PROVISIONING_ENABLED env (default true); if false, skip (kill switch)

2. **src/lib/provisioning/service.ts** — processOneJob(jobId):
   - Load job with order, plan, machineProfile
   - Idempotency: if ProvisionedVm exists with vultrInstanceId, skip (already provisioned)
   - Transition: payment_confirmed → provisioning_requested → provisioning
   - Set provisioningStartedAt when entering provisioning
   - Get vultrPlanId, vultrRegion from MachineProfile; if missing, set failureCode='MISSING_PROFILE', failureMessage, status=failed
   - Call vultr.createInstance({ region, plan: vultrPlanId, os_id: Windows GPU image ID, label: `cloudpc-${orderId}`, hostname })
   - On success: store vultrInstanceId, ipAddress from response
   - Poll vultr.getInstance(id) every 30s until status === 'active' (max ~15 min)
   - When active: set readyAt, status=vm_ready, compute expiresAt = readyAt + plan.durationHours (hours)
   - On any error: set failureCode, failureMessage, lastProviderResponse (JSON of error), retryCount++, nextRetryAt = now + backoff
   - Use connectionMethod='parsec', connectionState='ready' when vm_ready
   - VM-04: Use Vultr Windows GPU image (os_id from Vultr docs — e.g. 215 for Windows 2022). Parsec pre-installed via custom image if available; otherwise document that image must have Parsec. For MVP use standard Vultr Windows GPU image; add startup script placeholder for future Parsec config if needed.

3. **src/app/api/cron/process-provisioning/route.ts** — GET handler:
   - Verify cron secret (CRON_SECRET header) to prevent public abuse
   - Call processProvisioningJobs()
   - Return 200 { processed: N }
