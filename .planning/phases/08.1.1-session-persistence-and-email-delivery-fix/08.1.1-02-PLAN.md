<objective>
Add middleware-based session path as fallback for Next.js 15 cookies() issues. Middleware has direct request access; getSessionFromCookies uses header when set. Also surface email_failed to user on login page.
</objective>

<tasks>

### Task 1: Create middleware
- Create `src/middleware.ts` (Next.js convention: root or src)
- Import `jwtVerify` from jose, `NextResponse` from next/server
- Match routes: `/dashboard`, `/orders`, `/profile`, `/billing`, `/admin`, `/plans` (for session check on protected pages)
- For matched routes: read `session` cookie from request
- If cookie present: verify JWT with SESSION_SECRET (same as session.ts)
- If valid and not expired: clone request headers, set `x-session-id` with sessionId from payload
- Return `NextResponse.next({ request: { headers: requestHeaders } })` so server receives the header
- If cookie absent or invalid: return `NextResponse.next()` (no header; getSessionFromCookies uses cookies() path)
- Use same getSecret() logic as session.ts; handle missing SESSION_SECRET gracefully (skip middleware verify in that case)

### Task 2: getSessionFromCookies header fallback
- Import `headers` from `next/headers`
- At start of `getSessionFromCookies()`: `const h = await headers(); const sessionIdFromHeader = h.get('x-session-id');`
- If sessionIdFromHeader: do DB lookup only (skip cookie read); return session if found and not expired
- Else: existing cookies() path

### Task 3: Login page email_failed handling
- Login page receives `email_failed` from searchParams (in addition to signup, email)
- Pass `emailFailedParam` to LoginForm
- LoginForm shows: "Não foi possível enviar o e-mail de verificação. Verifique RESEND_API_KEY no servidor ou solicite reenvio abaixo."

</tasks>

<verification>
1. With valid session cookie, navigate to /dashboard — middleware sets header; getSessionFromCookies returns session via header path
2. With invalid/expired cookie — no header; cookies() path returns null; redirect to login
3. Signup with RESEND_API_KEY unset → login?signup=1&email_failed=1 → user sees hint
</verification>

<notes>
- Middleware runs on Edge; jose works on Edge
- Header x-session-id is internal; not exposed to client
- If cookies() works, both paths yield same result; header path is faster (no cookie parse)
</notes>
